# Salesforce Lightning — Automation Ruleset (Mandatory)

This document defines **non‑negotiable** rules for Robot Framework tests against **Salesforce Lightning** (Aura + LWC). All new keywords, locators, and AI‑generated tests **must** comply. The goal is **resilience** to DOM churn, **deterministic** waits (no timing guesses), and **safe** interaction with LWC, iframes, and hybrid Classic surfaces.

---

## 1. Locator strategy

### 1.1 Default preference order

Use locators in this order **unless** a rule in §1.2 forces otherwise:

1. **Stable Salesforce / SLDS semantics**
   - `data-target-selection-name`, `data-field-name`, `data-id`, `data-value` (on combobox items where applicable)
   - `aria-label`, `aria-labelledby`, `role`, `name` (when generated by Salesforce and stable for the scenario)

2. **Test hooks (strongly preferred when your org adds them)**
   - `data-qa-locator`, `data-testid`, `data-cy`, or org‑standard QA attributes on **wrapper** components

3. **LWC / Lightning component tags in the light DOM**
   - `lightning-button`, `lightning-input`, `lightning-base-combobox`, `lightning-datepicker` — target **published** attributes (`title`, `label`, `variant`) when exposed on the host element **or** on child nodes that Selenium can see without piercing

4. **Title / visible text on interactive nodes**
   - `//button[@title='Save']` — acceptable when `title` is the primary contract for that control

5. **Scoped CSS (short, semantic)**
   - Prefer `css:button.slds-button_brand` **inside a known container** (e.g. modal footer) over page‑wide XPath

### 1.2 Forbidden or restricted patterns

| Pattern | Rule |
|--------|------|
| **Long XPath chains** with many `//div[contains(@class,'slds-…')]` segments | **Do not add** new ones; refactor to shorter scope + `aria-label` / `data-*` / component tag |
| **Absolute XPath** from `/html/body/...` | **Forbidden** |
| **XPath index hacks** like `[1]` for “first row” **without** a business reason | **Avoid**; use list semantics, `data-row-key`, or explicit row identity |
| **Copy‑pasted full XPath** from DevTools | **Forbidden** — treat as technical debt |

### 1.3 LWC Shadow DOM

- **Assume** many controls render **shadow roots**. Standard Selenium locators **cannot** see inside **closed** shadow trees from outside.
- **Rules:**
  1. Prefer interacting via **light DOM** parents (e.g. `lightning-input` host sometimes exposes an internal `input` in light DOM depending on version) — **verify** per Salesforce release.
  2. When interaction must target **shadow‑internal** nodes, use **one** of:
     - **`Execute Javascript`** with `element.shadowRoot.querySelector('…')` and then `.click()` / `.value = …` (centralize in **reviewed** helper keywords — no ad‑hoc JS in generated tests).
     - **Piercing selectors** only where Salesforce documents or team standards allow (e.g. specific `::part()` if applicable — rare).
  3. **Never** scatter one‑off shadow scripts; add a **named keyword** (e.g. `Click Lightning Button By Label`) and reuse.

### 1.4 Classic / Visualforce / hybrid pages

- If the page is **VF or Classic** inside Lightning, **iframe** rules (§3) apply before applying §1.1–1.3 to inner content.

## 2. Wait strategy

### 2.1 Forbidden

| Anti‑pattern | Rule |
|---------------|------|
| **`Sleep`** with fixed duration for UI readiness | **Forbidden** in new code (including “small” sleeps like `0.3s`). Existing sleeps are **debt** to remove. |
| **Implicit wait as substitute** for business conditions | Do not rely on global implicit wait alone for “page ready” |

### 2.2 Required patterns

1. **Explicit waits** via SeleniumLibrary:
   - `Wait Until Element Is Visible`
   - `Wait Until Element Is Not Visible`
   - `Wait Until Page Contains Element`
   - `Wait Until Keyword Succeeds` (retry bounded micro‑steps)

2. **Lightning / SLDS busy indicators** — wait for **absence** or **stable state** before click/type:

   Examples (org may extend the list in `GlobalLocators.robot`):

   - `div.slds-spinner_container`, `div.slds-spinner`, `.slds-spinner_container.slds-grid`
   - `lightning-spinner`
   - `forceChangeRecordType` overlay when it blocks primary actions
   - List / utility bar search spinners tied to list views

3. **Semantic “transaction” boundaries**

   After actions that trigger server round‑trips (Save, Next, Convert):

   - Wait until **blocking spinners** in the **active** modal or **main** column are gone.
   - Then wait for the **next** expected UI state (toast, redirect, modal close, field visible).

4. **Timeouts**

   - Use **explicit timeouts** on waits (e.g. `timeout=30s`), not unbounded except for human‑in‑the‑loop (MFA).

### 2.3 Centralization

- **New** “wait for Lightning idle” behavior must be implemented as **reusable keywords** (e.g. `Wait For Lightning Spinners Absent In Scope`) with documented scope (full page vs modal vs list view).

## 3. iFrame handling

### 3.1 Detection

- Before interacting, determine whether the target lives in the **top** document or in an **iframe** (Classic console, some VF, Email‑to‑Case, legacy embeds).

### 3.2 Rules

1. **Select Frame** using a **stable** locator for the iframe (`css:iframe[name='...']`, `id`, `title`, or `src` fragment if stable).
2. **Always pair** with **`Unselect Frame`** (or return to **default**) in:
   - **Teardown** / `Run Keyword And Finally`, or
   - Immediately after the **atomic** operation that required the frame — so subsequent steps do not run in the wrong context.
3. **Nested iframes** — document the **order** (outer → inner) in the keyword docstring; one keyword per nesting level if needed.
4. **Lightning default** — most **standard Lightning** UIs are **not** in iframes; **do not** assume iframe without verification.

### 3.3 Cross‑context locators

- Locators used **after** `Select Frame` must be valid **inside** that frame’s DOM. Do not reuse top‑level XPath that only worked outside the frame.

---

## 4. Compliance & reviews

- **Code review** for new keywords: check §1–§3.
- **AI‑generated** `.robot` files must be reviewed in the **Human‑in‑the‑Loop** editor; reject brittle XPath and raw `Sleep`.
- **Refactors** should migrate legacy XPath and sleeps incrementally; track in your backlog.

---

## 5. Versioning

- **Ruleset version:** attach Git tag or doc date when changing this file.
- Breaking locator changes in Salesforce (major releases) may require **targeted** updates to `GlobalLocators.robot` and shared keywords — not one‑off test patches.
